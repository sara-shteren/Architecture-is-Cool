מדריך למפתח — v1

Architecture-is-Cool

צ'אט RAG ללימוד דזיין פטרנס וארכיטקטורה — עם בדיקת הבנה אמיתית מול מקור, לא רק תקציר טקסט. המסמך הזה נותן את התמונה הגדולה לפני שיושבים על הקוד.

1. מה בונים, בפסקה אחת

המשתמש (מפתח מנוסה) בוחר נושא מכפתורים (למשל "Strategy Pattern", "CQRS") — או מדביק URL של מאמר משלו והמערכת קולטת אותו חי. לכל נושא יש שלוש דרכים קלילות להבין (בדיחה / אנלוגיה / קוד לפי רמה), אבל הלב של המוצר הוא מצב מבחן: הצ'אט שואל שאלה מעוגנת בקטע המקור, המשתמש עונה בטקסט חופשי, וסוכן Critic מעריך את התשובה מול המקור עצמו — לא מול "ידע כללי" של המודל. בסוף מוצע נושא קשור שעוד לא נלמד.

למה זה לא "ChatGPT + PDF": שם אין זיכרון בין הפעלות ואין דרך לדעת אם באמת הבנת. כאן יש קורפוס מצטבר, מעקב התקדמות, והערכה שתמיד מעוגנת בקטע מקור ספציפי. לולאת הדרכה + בדיקת הבנה היא המוצר — לא הסיכום.

3. איך המערכת בנויה — Modular Monolith

Deployment אחד (קונטיינר FastAPI יחיד), אבל גבולות מודול נאכפים בקוד: routes לא נוגעים ב-DB/LLM ישירות — רק דרך services; agents לא נוגעים ב-DB ישירות — רק דרך services/vector. שרשרת תלות חד-כיוונית. זה מה שמאפשר לפתח דרישות שונות במקביל בלי דריכה זה על זה.

למה לא מיקרוסרוויסים? תשתית מיותרת לפרויקט צד. למה לא מונוליט "רגיל"? בלי גבולות אכופים אי אפשר להוכיח שזו באמת מערכת מולטי-אייג'נט ולא "פונקציה אחת גדולה". Modular Monolith נותן את שני העולמות.

4. לולאת הליבה — Multi-Agent Graph (LangGraph)

הזרימה מנוהלת כגרף מפורש עם state משותף (TopicState), לא if/else מפוזרים. ה-Critic (כתום) הוא לולאת האנטי-הזיות: סוכן שבודק פלט של סוכן אחר מול מקור.

כלל חשוב ל-Critic: אם המשתמש עונה נכון אבל קטע המקור פשוט לא מכסה את זה — הפסק הוא unclear עם תשובה כנה ("המקור לא מתייחס לזה — הנה מה שהוא כן אומר"), אף פעם לא incorrect שגוי. מפתחים מנוסים עונים מניסיון מעבר למאמר — זה תרחיש צפוי, ופסילה שלו תהרוס את האמון בכל הלולאה.

5. קליטת URL של משתמש (F1.5) — איך זה עובד

אותו pipeline בדיוק כמו קורפוס ההתחלה (fetch ← חילוץ טקסט ← חיתוך ← embedding ← שמירה), רק שהטריגר הוא בקשת HTTP במקום סקריפט. אין קוד נפרד.

  1. POST /topics/from-url — נוצרת שורת topic עם status='processing' ומזהה חוזר מיד. הקליטה רצה כ-background task של FastAPI (בלי תור/worker).
  2. הפרונט עושה polling על GET /topics/{id} ומציג חיווי טעינה עד שהסטטוס מתחלף.
  3. בדיקת cache קודם כל (hash של התוכן): אם ה-URL הזה כבר נקלט פעם — מדלגים על כל העיבוד ומקשרים לתוכן הקיים. הגשה כפולה עולה fetch אחד, כלום מעבר.
  4. כותרת הנושא נגזרת אוטומטית: <title> או <h1> של הדף; אם אין — סיכום שורה אחת מה-LLM.
  5. בסיום: status='ready' — ומכאן הנושא זהה לחלוטין לנושא אוצר: אותה לולאה, אותו גרף, אותו מאגר המלצות. כישלון (URL שבור, דף ריק) ← status='failed' עם סיבה ברורה. אין מצב ביניים תקוע.
גבולות v1: בדיקות קלט בסיסיות בלבד — סכימת http(s), תקרת גודל ~2MB, timeout ל-fetch. סינון תוכן/מודרציה/מכסות פר-משתמש — v2 במוצהר.

6. סכמת DB (Postgres + pgvector, הכל במסד אחד)

CACHEsource_files — תוצרי עיבוד לפי content_hash + pipeline_version. פגיעה בקאש = מדלגים על כל העיבוד

DOMAINtopics — קישור נושא ← מקור. כולל chunk_range, topic_embedding, ו-status/source (בשביל F1.5)

RAGchunks — יחידת השליפה, embedding לכל קטע. בלי topic_id — המיפוי נושא↔קטעים חי רק ב-chunk_range (מקור אמת יחיד)

SQL רגילuser_progress — הושלם/לא + ציון. אפס embeddings

Auditquiz_attempts — כל ניסיון מבחן, כולל ה-retries של ה-Critic

CACHEgeneration_cache — תוצרי LLM (בדיחות/אנלוגיות/קוד/שאלות) לפי (topic, mode, level, prompt_version) — ראו סעיף 7

ההפרדה שהכי חשוב לשמור: user_progress זה lookup ("הושלם? כן/לא"), ומנוע ההמלצות זה דמיון סמנטי. אלה שני עולמות — אסור לערבב, וזה גם ההסבר הכי נקי בראיון.

7. עלויות — קאש עצלן, לא הכנה מראש

העיקרון: שומרים מה שכבר קרה, לא מכינים את מה שעוד לא ביקשו.

  1. המשתמש הראשון שמבקש שילוב מסוים (נושא × מצב × רמה) מקבל יצירה חיה אמיתית מה-LLM — והתוצאה נשמרת ב-generation_cache.
  2. עד 3 גרסאות נשמרות לכל שילוב: כל עוד יש פחות מ-3 — קוראים ל-LLM ומוסיפים; מ-3 ומעלה — בוחרים גרסה אקראית קיימת, בלי LLM בכלל.
  3. הרוטציה בין גרסאות מונעת את התחושה של "אותה בדיחה מילה במילה" — גם המשתמש המאה מרגיש תוכן חי.
  4. הקריאה היחידה שלעולם לא נשמרת בקאש: הערכת ה-Critic — כל תשובה ייחודית. זה גם הרגע שהכי מוכיח "AI אמיתי" בדמו.

התוצאה: דמו של ~100 משתמשים עולה פחות מדולר (מעבר זמני ל-tier בתשלום עם תקרת תקציב ביום הדמו). המלצות הנושא-הקשור הן שאילתת pgvector — חינם, בלי קאש בכלל.

8. אבטחה — שלושה שערים

1. API Key + Rate Limiting. מפתח סטטי ב-header (X-API-Key), בדיקה ב-dependency אחד על כל ה-routes חוץ מ-/health. ה-rate limiting (slowapi) חייב להיות פר-מפתח/session, לא רק גלובלי — אחרת משתמש אחד שורף את המכסה של כולם.

2. Domain-Scoping Guard. לפני שהודעת טקסט חופשי מגיעה ל-LLM, בודקים דמיון סמנטי מול קורפוס הנושאים. מתחת לסף ← "העוזר עונה רק על ארכיטקטורה ודזיין פטרנס", בלי לקרוא ל-LLM בכלל. מונע שימוש בצ'אט כפרוקסי חינמי ל-Gemini. חל רק על קלט חופשי — לחיצות כפתור לא צריכות בדיקה.

3. סודות. הכל דרך env (core/config.py), אף מפתח בקוד, .env ב-gitignore מהקומיט הראשון, לוגים בלי תוכן פרומפטים.

הסף של ה-guard (0.55) הוא placeholder — לפני נעילה מכיילים אמפירית: סקריפט שמריץ ~20-30 שאלות בתחום ומחוץ לתחום ומראה את התפלגות הציונים. הערך יושב ב-env, לא בקוד.

10. סדר עבודה — במה מתחילים

הפירוק המלא נמצא ב-docs/TASKS.md — משימות מנוסחות כדרישות עסקיות לפי נושא, כל נושא עם הגדרת "גמור כאשר" בשפת התנהגות. סדר הנושאים שם הוא סדר העבודה:

  1. הקמת התשתית — Docker, config, DB, ממשקי Provider, הקפאת TopicState. גמור כש-docker compose up עובד על מכונה נקייה.
  2. קליטת ידע — צנרת ה-ingestion + קורפוס ההתחלה, עם מנגנון שמונע עיבוד כפול ושריפת מכסה.
  3. נושאים + הוספת URL ע"י משתמש — יחידת הלמידה, ומעליה F1.5.
  4. הליבה — שלד גרף הסוכנים, מצבי ההסבר, המבחן וה-Critic, שאלות חופשיות.
  5. התקדמות והמלצות — מעקב השלמה + הצעת הנושא הבא.
  6. רוחבי — קאש היצירה, שומר-התחום, אבטחה.
  7. UI — ממשק הצ'אט שמרכז הכל, עד שכל תרחישי ה-PRD מודגמים מהדפדפן.
  8. טסטים לאורך כל הדרך — כל נושא נמסר עם הטסטים שלו, לא באצ' בסוף.

11. מה במפורש לא ב-v1

  • קרולר אוטונומי (מעקב אחרי לינקים, אינדוקס אתרים שלמים) — רק seed + URL בודד של משתמש.
  • מודרציה/הקשחה לקלט URL פתוח — v2.
  • קליטת PDF — נקודת הרחבה מתועדת (ענף חילוץ חדש), לא ממומשת.
  • ריבוי משתמשים/חשבונות אמיתי — מפתח API סטטי אחד (דמו single-tenant).
  • גיימיפיקציה מעבר למעקב השלמה/שליטה.
  • F10 מצב ראיון — נדחה ל-v2 (התכנון שמור).

12. הגדרת "עובד" (Success Criteria)

  • זרימה מלאה בדמו: בחירת נושא ← שלושת מצבי ההסבר ← מבחן ← תשובה שגויה ← הסבר מעוגן עם ציטוט ← הצעת נושא קשור.
  • בחירה חוזרת בנושא שהושלם מציעה "רענון או נושא חדש קשור" — מוכח חי.
  • הדבקת URL של מאמר אמיתי מייצרת נושא עובד תוך ~30 שניות, עם חיווי טעינה וכותרת אוטומטית.
  • git clone ← docker compose up ← UI עובד, בלי תיקונים ידניים.
