# תקציר — Architecture-is-Cool

## הרעיון

צ'אט מבוסס RAG שהופך חומר כבד (ספרי דזיין פטרנס, ארכיטקטורה) לחוויית למידה מהירה, מבוססת
נושאים — עם בדיקת הבנה אמיתית, לא רק תקציר טקסט.

המשתמש **בוחר נושא** (למשל "מונוליט", "Strategy Pattern"). הצ'אט עונה מעוגן בקורפוס ידע
ציבורי (מאמרים/מדריכים, לא ספר קנוי), מציע שלוש דרכים קלילות להבין את הרעיון (בדיחה,
אנלוגיה, קוד לפי רמה), אך המרכיב המרכזי הוא **מצב מבחן**: הצ'אט שואל שאלה, המשתמש עונה,
והמערכת מעריכה את התשובה **מול קטע המקור בפועל** — לא מול "ידע כללי" של המודל. תשובה
שגויה מוסברת עם ציטוט מדויק; תשובה נכונה יכולה להוביל להצעת נושא קשור שעדיין לא נלמד.

**למה זה שונה מ"ChatGPT + PDF מועלה":** שם התשובה מבוססת על מסמך בודד שהועלה עכשיו, בלי
זיכרון בין הפעלות ובלי דרך לדעת אם המשתמש באמת הבין. כאן יש שליפה (retrieval) על קורפוס
מצטבר, מעקב אחרי מה שכבר נלמד, והערכה תמיד מעוגנת בקטע מקור ספציפי — הלולאה של
"הדרכה + בדיקת הבנה" היא המוצר, לא רק סיכום טקסט.

## יכולות עיקריות (F1-F9, בקצרה)

| # | יכולת |
|---|---|
| F1 | בחירת נושא ← שליפה דטרמיניסטית של קטעי המקור שלו |
| F1.5 | המשתמש מדביק URL של מאמר שהוא רוצה ללמוד ← המערכת קולטת אותו חי (~10-30 שניות עם חיווי טעינה), מזהה כותרת אוטומטית, והנושא החדש מקבל את כל הלולאה (בדיחה/אנלוגיה/קוד/מבחן) — **בנוסף** לכפתורי הנושאים הקבועים, לא במקומם |
| F2 | שלושה מצבי הסבר: בדיחה, אנלוגיה, קוד (לפי רמה: ג'וניור/סניור/ארכיטקט) |
| F3-F4 | מבחן מבוסס-מקור: שאלה מעוגנת ← הערכת תשובת המשתמש מול קטע המקור (לא שיפוט כללי) |
| F5 | הצגת רפרנסים נוספים מהקורפוס כשרלוונטי (multi-hop retrieval) |
| F6 | שאלות חופשיות בתוך נושא — שליפה מסוננת לנושא הנוכחי |
| F7 | מעקב התקדמות (SQL רגיל בלבד — **לא** embeddings, ראו הבהרה למטה) |
| F8 | המלצת נושא הבא, מבוססת דמיון סמנטי (embeddings) בין נושאים, מסוננת למה שלא נלמד עדיין |
| F9 | מייל תזכורת קליל — טיזר + שאלת מבחן אחת, לא תקציר יומי מלא |

## זרימה בסיסית

```
משתמש בוחר נושא
   → שליפת קטע מקור (deterministic, לפי טבלת קישור topics)
   → משתמש בוחר מצב: בדיחה / אנלוגיה / קוד
   → משתמש נכנס למצב מבחן
        → הצ'אט שואל שאלה מעוגנת בקטע
        → משתמש עונה
        → Critic מעריך מול המקור (לא מול ידע כללי)
        → תשובה שגויה: הסבר + ציטוט; אפשרות retry (עם עצירת לופ)
        → תשובה נכונה: אופציה לרפרנסים נוספים מהקורפוס
   → לאחר סיום/re-entry: הצעת נושא קשור (embeddings) שעוד לא נלמד
```

## עלויות ודמו בקנה מידה (עד ~100 משתמשים)

בחירת נושא היא דרך כפתורים (רשימה מוגדרת מראש), אבל התוכן **לא** מוכן מראש — הוא נוצר
חי בפעם הראשונה שמישהו מבקש שילוב מסוים (נושא × מצב × רמה), ואז נשמר ב-cache. כדי שזה
לא ירגיש "קפוא" (אותה בדיחה מילה במילה לכל מי שילחץ), נשמרות כמה גרסאות לכל שילוב
ונבחרת גרסה אקראית — כך שגם המשתמש ה-50 מקבל תחושה של תוכן חי, לא תשובה משוננת. המלצות
הנושא-הקשור הן שאילתת DB (pgvector), לא קריאת API — אין מה לחסוך שם. הקריאה
היחידה שתמיד חיה לגמרי היא **הערכת ה-Critic** לתשובה חופשית (ייחודית לכל משתמש, אי
אפשר לקאשש). ליום דמו עמוס: מעבר ל-tier בתשלום עם תקרת תקציב — עלות בפועל של פחות
מדולר ל-500 הערכות. פירוט ב-ARCHITECTURE.md §8.

## הבהרה חשובה: שתי שכבות שונות, לא לערבב

- **`user_progress`** — טבלת SQL רגילה (מה נלמד, מתי, כמה טוב). זו שאלת "כן/לא הושלם",
  לא שאלת דמיון סמנטי. **אין כאן embeddings בכלל.**
- **מנוע ההמלצות (`suggest_related_topic`)** — כאן כן embeddings: דמיון סמנטי בין נושאים,
  מסונן לפי מה שכבר ב-`user_progress` כ"הושלם". זהו הרכיב היחיד במערכת שמשתמש ב-retrieval
  לצורך **המלצה**, לא רק לצורך מענה על שאלה.

הפרדה נקייה בין השתיים היא מה שהופך את ההסבר בראיון לפשוט וברור: "יש לי state רגיל, ויש
לי retrieval-based recommendation — הם לא אותו דבר ולא צריכים להיות."

לפרטים טכניים מלאים (schema, ממשקים, גרף ה-agents, פריסה) ראו את המסמכים באנגלית:
`docs/PRD.md`, `docs/ARCHITECTURE.md`.
